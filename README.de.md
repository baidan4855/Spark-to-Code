# Spark to Code

Sprache: [English](README.md) | [中文](README.zh-CN.md) | [Français](README.fr.md) | **Deutsch**

Dieses Repository ist kein Quellcode-Repository.  
Es ist ein Kollaborationsraum, der Ideen, Probleme und Entwickler zusammenbringt.

Das Ziel ist einfach: Gute Ideen sichtbar machen, validieren und umsetzen.

## Zweck

- Für Ideengeber: reale Pain Points teilen, bevor Ideen in privaten Notizen verschwinden.
- Für Entwickler: sinnvolle Side Projects und Mitstreiter finden.
- Für die Community: bestehende Lösungen früh erkennen und Doppelarbeit vermeiden.

## Was dieses Repository bietet

1. Request-Hub  
Jede Person kann konkrete Software-Bedürfnisse und Pain Points einreichen.

2. Team-up-Bereich  
Ideengeber können Mitwirkende suchen und Rollen/Tech-Stack beschreiben.

3. Build-Updates  
Teams können Meilensteine, Blocker und Fortschritt in der Entwicklung teilen.

4. Tool-Discovery  
Wenn bereits eine reife Lösung existiert, kann die Community direkt darauf verweisen.

## Beteiligungsmodell: Issue-first, PR-second

### Issue-first

Issues sind der Hauptkanal:
- `idea`: neue Idee oder Problem einreichen
- `looking-for-solution`: Empfehlung für bestehende Software anfragen
- `building-update`: Fortschrittsupdate zu laufendem Projekt
- Kommentare: Scope schärfen, Tools empfehlen, Team beitreten

### PR-second

Pull Requests sind optional und dienen nur der Repository-Pflege:
- Dokumente und Templates aktualisieren
- kuratierte Listen wie `PROJECTS.md` pflegen

PRs sind nicht der primäre Weg, Ideen einzureichen.

## Wer darf posten

Alle.  
Dieses Repository ist für Entwickler und Nicht-Entwickler offen.

## Community-Regeln

1. Für neue Issues bitte die Vorlagen nutzen.
2. Unklare Ideen vermeiden; Szenario und Constraints konkret beschreiben.
3. Wenn es eine reife Alternative gibt, offiziellen Link teilen und Schließen vorschlagen.
4. Wer einem Team beitreten möchte, nennt Rolle, Stack und verfügbare Zeit.
5. Hier veröffentlichte Ideen gelten als öffentlich geteilt. Keine vertraulichen Informationen posten.

## Label-System (Englisch)

| Label | Bedeutung |
| --- | --- |
| `idea` | Neue Idee oder Pain Point |
| `looking-for-solution` | Suche nach bestehenden Tools |
| `team-wanted` | Mitwirkende gesucht |
| `building` | Aktiv in Entwicklung |
| `solved` | Durch Tool gelöst oder implementiert |
| `duplicate` | Duplikat eines bestehenden Issues |
| `needs-info` | Wichtige Informationen fehlen |
| `stale` | Lange ohne Aktivität |

## Leichter Maintainer-Workflow

1. Wöchentliches Triage: deduplizieren, labeln, gelöste Threads schließen.
2. Teams dazu anhalten, Fortschritt im selben Issue zu aktualisieren.
3. Unvollständige Issues mit `needs-info` markieren und nach langer Inaktivität schließen.
